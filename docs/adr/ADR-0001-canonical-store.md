# ADR-0001: Canonical store invariant

**Status:** Accepted
**Date:** 2026-07-12

## Context

The repository contradicted itself about what "canonical" meant for memory. `zeref/db.py` and `zeref/memory/graph.py` treated Markdown as the source of truth, with SQLite as a derived snapshot. `zeref/memory_state.py` and `zeref/memory/core.py` treated SQLite as canonical, with Markdown as a generated view. Both halves of the codebase could not simultaneously be correct, and downstream code (migrations, sync, audits) inherited whichever assumption its author happened to hold.

## Decision

Resolve the contradiction with one invariant, applied everywhere:

- **SQLite** — canonical current state.
- **JSONL** — canonical append-only history (hash-chained, replayable, schema-validated).
- **Markdown** — generated human-readable view. Never hand-edited as a source of truth; every generated file carries a `<!-- GENERATED BY ZEREF. SOURCE: memory/state/zeref.sqlite. DO NOT EDIT DIRECTLY. -->` header.
- **TOON** — optional generated model-input view, selected by codec benchmarks, never canonical.
- **Parquet** — optional generated analytical/benchmark export, never canonical.

State must always be rebuildable from JSONL alone. Markdown and TOON must always be demonstrably derived, never authored by hand as the record of truth.

## Consequences

- `zeref/db.py` and `zeref/memory/graph.py` must be corrected (or replaced) to stop treating Markdown as canonical; `zeref/memory_state.py` and `zeref/memory/core.py`'s SQLite-canonical assumption is the one that survives.
- Any tooling, doc, or importer that currently writes Markdown and expects it to persist as the record of truth must be migrated to write SQLite + JSONL first and regenerate Markdown after.
- Implementation (versioned SQLite migrations under `zeref/migrations/`, the JSONL v2 event envelope, and the Markdown view generator) lands in PR 2 (`zeref/storage/`, `zeref/migrations/`) per the vNext architecture plan §6 and §20. This ADR records the decision; PR 2 is where it becomes enforced code.
- Existing memory cards, atom JSONL files, and `DECISIONS.md`/`CONFLICTS.md`/`RISKS.md`/`MEMORY.md` need a dry-run-capable, idempotent importer with backup and rollback before this invariant can be enforced against pre-existing data.
